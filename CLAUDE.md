## What this is

COLT — an AI-powered youth athlete development platform (training, gamification,
club engagement, AI coaching). Scaffolded from the planning documents in `docs/`. **Built so
far (Supabase):** auth, the multi-tenant data model, club onboarding, athlete management, the
Training Library, the full Gamification engine (XP → Tiers → The Ladder → Caps → Heat), and the
Matchday Challenge (Weekly Programs — the weekly "Set"). The remaining product modules (AI Coach,
Payments) are not yet built — and per the freemium rollout below, the paywall/Payments is
deferred to Phase 2 (Phase 1 ships everything free).

## Commands

Node 24 LTS and npm are required (installed via winget at `C:\Program Files\nodejs`). If `node`
isn't found, open a new terminal so the updated PATH is picked up.

```bash
npm run dev     # start dev server (Turbopack) at http://localhost:3000
npm run build   # production build + TypeScript typecheck
npm run start   # serve the production build (run build first)
npm run lint    # ESLint (eslint-config-next, flat config in eslint.config.mjs)
```

There is **no test runner configured yet** — no Jest/Vitest/Playwright. Add one before writing
tests, then document the command here. Typechecking currently happens only as part of `npm run build`.

**Supabase setup:** copy `.env.example` → `.env.local` and fill in a real project's URL + anon
key (`.env.local` currently holds placeholders so the app builds/runs without a backend). Until
real credentials are set, sign-up/login will fail at runtime but every page still renders and
protected routes still redirect.

The Supabase CLI is a dev dependency (run via `npx supabase`; `supabase/config.toml` is the
project config). Apply migrations one of two ways:

```bash
# Remote: link once, then push the migrations in supabase/migrations/
npx supabase login                         # needs a personal access token
npx supabase link --project-ref <ref>      # needs the DB password
npm run db:push                            # supabase db push

# Local (requires Docker): full local stack + auto-applied migrations
npm run db:start    # supabase start — prints local URL + anon key for .env.local
npm run db:reset    # re-apply all migrations from scratch
```

Other scripts: `db:diff` (generate a migration from schema changes), `db:migrate <name>` (new
empty migration), `db:stop`. Migrations are timestamp-prefixed (`<YYYYMMDDHHMMSS>_name.sql`) —
the format `supabase migration new` expects; apply them in filename order. You can still paste a
migration's SQL into the dashboard SQL editor instead.

## Layout

- `src/app/` — App Router. Route groups: `(auth)` (login/signup + auth Server Actions in
  `actions.ts`), `(dashboard)` (protected; layout loads the profile, redirects no-club users to
  `/onboarding`, and renders role-based UI). Protected pages live under `(dashboard)/dashboard/`:
  `/dashboard` (role dashboard) and `/dashboard/training` (Training Library, filterable by sport +
  age group via URL search params). `/onboarding` (standalone protected route) lets a user create a
  club (becoming `club_admin`) or join one by share code.
- `src/lib/supabase/` — Supabase clients: `client.ts` (browser), `server.ts` (server, async
  `cookies()`), `proxy.ts` (session refresh + route guard), `config.ts` (env access).
- `src/lib/types.ts` — domain models + the `Database` type for the typed client.
- `src/lib/auth.ts` — `getCurrentUser()` (user + profile + club). `src/lib/validation.ts` — zod schemas.
- `src/proxy.ts` — Next.js 16 Proxy entry (the renamed middleware); delegates to `lib/supabase/proxy.ts`.
- `supabase/migrations/` — SQL schema (tables, enums, RLS policies, new-user trigger).
- `public/` — static assets. `docs/` — the three planning `.docx` (read by unzipping
  `word/document.xml`; `pandoc` is not installed). `AGENTS.md` — Next.js 16 rules (imported above).

## Data model & access control (built)

Tenancy is **club → athlete**, with every auth user owning one `profiles` row that carries their
`role` and `club_id`. RLS enforces club-scoped, role-based access in the database itself:

- `profiles`: a `handle_new_user()` trigger creates one on sign-up (role defaults to `parent`).
- `current_club_id()` / `current_role()` are `SECURITY DEFINER` SQL helpers used by policies to
  avoid recursive RLS. Reads are scoped to the caller's club; writes additionally gate on role.
- Onboarding writes go through `SECURITY DEFINER` RPCs (`create_club`, `join_club` in the
  `..._club_onboarding` migration), called via `supabase.rpc(...)`. They bypass RLS deliberately so a user with no club can
  create/join one without RLS exposing a way to enumerate other clubs. New RPCs must be added to
  both the SQL and the `Functions` block in `types.ts` to stay typed.
- **Typed-client gotcha:** domain models in `types.ts` must be `type` aliases, not `interface`s.
  supabase-js requires `Row extends Record<string, unknown>`, which interfaces don't satisfy —
  using an interface silently collapses all query result types to `never`. Likewise, use
  `Record<never, never>` (not `Record<string, never>`) for empty schema sections.
- **Training Library (`drills`):** a catalogue of training content tagged by `sport` + `age_group`
  (reusing those enums). `club_id IS NULL` = platform/global content visible to every club;
  non-null = club-specific. RLS follows the established pattern: **SELECT** visible when
  `club_id IS NULL OR club_id = current_club_id()` (admins see all); **INSERT/UPDATE** gated to
  `coach`/`club_admin`/`admin` for their own club only (global drills are admin/seed-only). Seeded
  with global drills across all three MVP sports (Rugby League, Soccer, Basketball) and all three age
  groups. Add new content tables this way.

## Product context (from docs/)

The business model was revised in June 2026 (see the validation docs in the project root:
`NextGen_Athlete_Validation_Report.docx`, `_Validation_Deck.pptx`, `_Financial_Model.xlsx`).
The original `docs/` planning files describe the **superseded** model — defer to the revised one:

- **Revenue (revised — free-to-club, 70/30):** clubs pay **no licence** and access the product
  free; athletes/parents pay $9.99/mo; the split is **70% COLT / 30% club**. The club is a
  zero-cost distribution channel that earns 30% for promoting the app (≈$84/yr per paying athlete
  to COLT, ≈$36 to the club). All revenue is subscription-driven, so **parent conversion is the
  headline KPI**, tracked per club. (Old model — club licences $2,500–$10,000 + 60/40 — is dead.)
- **MVP:** Rugby League, Soccer, and Basketball, three age groups. RL + Soccer were the validated
  launch pair; **Basketball was added as a first-class sport** (`sport` enum) for the QLD pilot —
  see the operating-model refinement below.
- **The moat is the club–parent–athlete engagement loop + gamification, NOT "AI coaching"**
  (which the validation flags as commoditizing). Prioritize the engagement loop accordingly.
- **Planned modules:** Auth & Profiles, Training Library, Weekly Programs, Gamification (XP,
  Leaderboards), AI Coach, Payments/Subscriptions, and role dashboards (Parent / Coach / Club).
- **Architecture is multi-tenant and role-based:** a club → athlete tenancy hierarchy with
  role-scoped access, and 70/30 subscription billing attributed to the club that drove the signup.
- **Stale field — REMOVED:** `clubs.license_tier` and the `license_tier` enum (from the dead
  licence-tier model) were dropped in the `..._remove_license_tier` migration; clubs are now free.

### Freemium model & go-to-market (decided June 2026)

The $9.99/mo subscription is delivered as **freemium**, with the free/paid line following a
**collective-vs-individual** split. The engagement loop + gamification is the moat, and loops
need squad-wide density — so everything that builds network effects is free, and monetization
sits on the individual athlete's edge layered on top.

- **Free — the collective layer (builds audience + the moat):** profiles, squad membership and
  join-by-code, the core/global Training Library, club communications and scheduling, the squad
  feed, and *social* gamification — earning XP, squad leaderboards, streaks (Heat), basic Tiers.
  This is also what the club promotes with zero friction. **Never paywall club comms or squad
  participation** — gating the loop kills the network effect and the club's reason to push it.
- **Paid — the individual layer (drives conversion), now split into two tiers (T1/T2 — see the curated-expert refinement below):** personalised Weekly Programs
  (the strongest recurring-value driver), full personal Ladder progression + individual
  challenges, the deeper/position-specific library, AI Coach (kept secondary per positioning),
  and parent insight dashboards (progress, attendance, development tracking). The payer is the
  parent, so parent-facing visibility is a primary willingness-to-pay lever; conversion is
  **per-athlete**, which is why parent conversion is the headline KPI.
- **Rollout — start free, build audience:** **Phase 1 (now)** ship everything free and don't
  build the paywall yet — the goals are landing whole clubs/squads, proving the loop, and
  gathering data on which features people lean on (that data decides where the paywall goes).
  **Phase 2** introduce premium around the most-used features (expected: Weekly Programs +
  personal progression + parent insights), grandfathering early adopters where reasonable.
  **Phase 3** marquee-athlete content (e.g. a Reece Walsh program) as a premium acquisition hook,
  bundled into the subscription.
- **Marketplace — RULED OUT.** A two-sided creator marketplace (athletes/coaches selling their
  own paid programs) is *not* the direction: cold-start dynamics, youth safeguarding/liability,
  and club channel-conflict make it a poor near-term fit. Star-athlete pull is **secondary** and
  lives in Phase 3 as a marketing/acquisition layer bundled into the sub — not a paid marketplace.

### Operating-model refinement — curated-expert hybrid + two paid tiers (June 2026 pilot)

A go-to-market refinement of the freemium model above, to be proven in ONE narrow beachhead before
generalising. Full detail in `COLT_QLD_Basketball_Pilot_Blueprint.docx` (project root).

- **Beachhead: Queensland basketball.** First live market is a single QLD basketball club — basketball
  is Australia's fastest-growing team sport (2nd most-played; ~1.3M players, 350k+ regular juniors; QLD
  210k+ participants with a SE-QLD court shortfall = demand > supply). RL, Soccer + Basketball are now
  all first-class sports in the app (`sport` enum + seeded basketball drills/skills); basketball is the
  live-pilot vehicle. Its standardised **benchmark battery** (the Tier-2 verification/data moat) is
  pilot scope and **not yet built**.
- **Curated expert SUPPLY, not a marketplace.** Clubs stay the (free) demand/distribution engine; COLT
  adds **one vetted expert coach** as the content supplier and owns the loop, billing and data in between.
  This fills the expertise gap volunteer-run clubs lack **without** the two-sided creator marketplace,
  which stays RULED OUT (cold-start, youth safeguarding, channel conflict). The expert is a *supplier,
  not a competitor* — COLT owns the parent relationship; the coach holds a WWCC + child-safe training and
  all contact is club-mediated (no 1:1 DMs/video).
- **Two paid tiers** (replacing the single $9.99 line; both keep the **70/30** club split; prices are
  proposed starting points to validate in-pilot):
  - **Tier 1 (~$9.99/mo) — the training edge:** tailored expert program + ongoing development (weekly
    drops, challenges) + full parent progress view.
  - **Tier 2 (~$19.99/mo) — "close to a personal coach":** everything in T1 **plus** verified benchmark
    testing, coach feedback / video review, and the development profile (player card, peer percentiles).
    Verification's operational cost sits behind the higher price deliberately.
- **Verification engine = the data moat.** Gamification is the *data-capture engine* (it drives logging),
  not the product. A standardised, sport-specific benchmark battery is captured **on video** (so it
  scales), validated by automation + coach sampling, and reported as **percentiles vs age/position peers
  over time**. Output is a **verified development profile** — the durable asset and the flywheel:
  credible longitudinal data attracts elite pathways (rep teams, academies, scouts), which pulls in the
  next wave of paying families.
- **Headline pilot KPI:** free → Tier 1 conversion, and Tier 1 → Tier 2 upgrade, per club. Go/no-go gates
  are in the blueprint. Clone the unit (club + expert program) sport-by-sport only after proof.

## Planned integrations — NOT yet installed

The Technical Blueprint specifies **Supabase** (auth + data — installed/wired), **Stripe**
(payments), and an **AI provider** (AI coach), deployed on **Vercel**.

- **AI Coach → use Claude (Anthropic), NOT OpenAI.** The blueprint named OpenAI, but that was a
  planning assumption, not a requirement — nothing was ever wired to it. This project standardises
  on **Claude via the official `@anthropic-ai/sdk`** (the TypeScript SDK; never raw HTTP). Calls are
  **server-side only** (Server Action / Route Handler — same write-path pattern as `actions.ts`), so
  the key never reaches the browser. Env var is **`ANTHROPIC_API_KEY`** (in `.env.local` /
  `.env.example`). Default model `claude-opus-4-8`; a cheaper tier (Haiku 4.5 / Sonnet 4.6) is a
  reasonable cost choice since the AI Coach is the *secondary, commoditizing* feature (not the moat)
  and ships **free in Phase 1**. Use adaptive thinking + streaming for the chat UX.
- **Stripe (Payments)** is deferred to Phase 2 (no paywall in Phase 1). When building it, implement a
  **two-tier** subscription (Tier 1 ~$9.99 / Tier 2 ~$19.99 — see the curated-expert refinement above),
  each with the **70/30** split (not 60/40), attributed to the referring club.

Neither `@anthropic-ai/sdk` nor `stripe` is in `package.json` yet — install and wire them up as
their modules are built.
